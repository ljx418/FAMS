# FAMS 当前投资系统说明与咨询材料

更新时间：2026-05-19

本文用于向 ChatGPT 或其他架构/量化/投资系统顾问咨询。重点说明当前系统目标、模块预期能力、已实现能力、差距和瓶颈。

## 1. 系统定位

FAMS 是一个个人/家庭投资资产管理与分析系统，当前最高优先级不是自动交易，而是“高可靠、高正确”的资产账本、行情刷新、持仓分析、告警、AI 辅助研究和选股回测。

核心原则：

- 查询结果必须正确：标的身份、资产类型、价格、净值、持仓、成本、盈亏和标签都要可追溯。
- 外部行情失败不能伪装成功。
- 本地旧价可以用于展示兜底，但不能算刷新成功。
- AI 建议必须基于结构化事实集和 evidenceRefs，不能让 LLM 直接猜。
- 选股策略必须可解释、可复现、可回测，并且胜率结论必须带样本量、基准和可信度。

## 2. 当前系统模块

### 2.1 资产与持仓管理

预期功能：

- 管理股票、ETF、基金、债基、黄金、现金等资产。
- 支持手工新增持仓、导入资产、编辑份额/市值/收益率。
- 非现金资产的现价/净值由系统查询，不允许人工篡改。
- 人工可录入当前总市值和收益率，系统反推份额、成本和盈亏。
- 持仓页、资产页、Dashboard 聚合口径一致。

已实现：

- `Asset Identity Resolver` 已落地，用于识别股票、ETF、基金、债基、现金、黄金等类型。
- 持仓支持新增买入，后端 `/api/v1/positions/manual-buy`。
- 支持按当前市值 + 收益率反推成本。
- 黄金、现金已修复为手工总额/金额口径。
- 基金/债基份额口径修复：`quantity = marketValue / currentPrice`。
- 股票/ETF 部分卖出成本按同花顺风格修复：卖出净回款冲减剩余持仓成本。
- 自动回归脚本：
  - `test:position-consistency`
  - `test:derived-quantity`
  - `test:transaction-cost`
  - `test:manual-buy-position`
  - `test:manual-assets`

Gap：

- 资产导入、手工编辑、行情刷新、分析建议虽然已接入 resolver，但整体流程还需要更多端到端测试。
- 组合级目标仓位、再平衡建议还较粗。
- 持仓研究面板已优化展示边界，但还没有逐只持仓自动调用完整股票事实集。

瓶颈：

- 资产类型一旦识别错误，会影响行情源、刷新口径和分析入口。
- 用户手工校准市值与系统刷新净值之间的边界必须持续严格校验。

## 3. 行情与净值可靠性

预期功能：

- 股票/ETF/基金/债基/黄金/现金分别使用正确行情源。
- 刷新价格必须拿到外部实时价或官方净值，失败要透明。
- 刷新后执行 post-refresh validation，防止错误行情污染数据库。
- 所有行情应显示来源、时间、置信度和 fallback 状态。

已实现：

- A 股和场内 ETF 优先走 Sina。
- 场外基金/债基优先使用东方财富官方净值 `eastmoney_nav`，天天基金估值只作为参考/兜底。
- 现金按 `currentPrice=1` 和金额口径维护。
- 黄金按元/克金价刷新，避免被误识别成股票。
- 价格刷新本地兜底不再计入成功。
- post-refresh validation 校验来源、异常跳变、资产类型和持仓公式。
- 前端资产页展示行情来源。
- 自动回归脚本：
  - `test:market-data`
  - `test:fund-nav-sources`

Gap：

- 历史 K 线缓存还不完善。
- 全 A 股选股每次实时拉取历史 K 线，耗时不可控。
- 还缺更稳定的多 provider 历史行情层，例如 AkShare/Tushare/Baostock/本地缓存。

瓶颈：

- WSL/本机代理环境下 Node axios 访问部分 HTTPS 源经常出现 400/502/timeout，因此很多地方改用了 curl fallback。
- 外部 provider 吞吐限制明显，全 A 扫描同步执行不可接受。

## 4. 告警与风控

预期功能：

- 支持持仓级止盈止损。
- 支持宽基指数回撤监控，例如上证指数回撤 10% 提醒建仓。
- 告警必须由可追溯的行情或持仓数据触发。

已实现：

- 持仓级止盈止损按收益率触发，例如 +5% / -5%，不是按价格。
- Alerts 页面支持风险告警展示。
- 宽基回撤监控已落地：
  - 默认上证指数、沪深300、中证500、中证1000、创业板指、科创50。
  - 默认 250 日观察窗口。
  - 默认 10% 建仓提醒。
  - 监控标的和阈值可配置。
- 自动回归脚本：
  - `test:stop-alerts`
  - `test:market-watch`

Gap：

- 告警策略还缺组合级优先级排序。
- 告警与交易计划的联动仍较弱。
- 需要更多历史场景回放测试。

瓶颈：

- 告警正确性依赖行情刷新正确性。
- 指数历史行情也需要稳定缓存。

## 5. 分析建议与持仓研究

预期功能：

- 总览、分析建议、AI 股票分析、AI 选股应合并成统一研究入口。
- 任意标的/板块搜索应返回消息面、基本面、技术面、支撑压力位、结合持仓的策略。
- 每条结论都必须有数据来源和证据引用。
- 数据不足时明确显示不足，不输出模板话术。

已实现：

- 分析建议页合并了 Dashboard AI 股票分析入口。
- 目标研究和 AI 股票分析接入 Asset Identity Resolver。
- 分析建议返回 `marketDataTrace`，前端展示行情来源、价格、置信度、时间、fallback 和 warnings。
- 持仓研究面板已优化：
  - 按股票、基金/ETF/债基、黄金、现金分别展示研究边界。
  - 前端显示市值、现价/成本、支撑/压力、止盈止损。
  - 基本面/技术面/消息面拆成三栏。
- 股票持仓会明确提示：完整基本面证据在股票分析事实集中，不在持仓面板里随意生成结论。

Gap：

- 持仓研究面板还没有逐只股票自动嵌入完整 `StockAnalysisFactSet`。
- 分析建议主链路仍有规则引擎成分，尚未完全升级为“结构化事实 + AI JSON 建议”。
- 板块研究仍较弱。

瓶颈：

- 如果对所有持仓实时生成完整三面分析，会引入大量外部 provider 请求，需要缓存、批处理和任务化。

## 6. 股票三面分析：基本面、技术面、消息面

预期功能：

- 技术面：外部成熟技术指标和评级，本地指标仅作审计。
- 基本面：估值、财报、成长、盈利质量、现金流、负债、ROE、行业分位、多源复核、公告原文。
- 消息面：新闻、公告、事件分类、情绪、影响方向、时效性。
- 汇总层必须输出 status、summary、evidenceRefs、blockedReasons。

已实现：

- `StockAnalysisFactSet` 已落地，schema 为 `stock.analysis.factset.v1`。
- 技术面：
  - 接入 TradingView Scanner 外部指标。
  - 展示综合评级、均线评级、振荡器评级、RSI、MACD、ATR、SMA 等。
  - 与 Eastmoney/Sina K 线做交叉校验。
  - 输出多源可信度评分。
  - `TechnicalAdviceModelRegistry` 第一版落地。
- 基本面：
  - 东方财富估值：PE、PB、总市值、流通市值。
  - 东方财富 F10 财报主指标：营收、归母净利、ROE、毛利率、资产负债率、经营现金流、EPS。
  - 行业分位：PE/PB/市值/ROE/资产负债率。
  - 东方财富同厂不同接口复核。
  - 搜狐证券重要财务指标页独立复核。
  - 公告原文定位：上交所 PDF 链接、披露日期、公告标题。
  - `601127` 当前基本面 facts 为 36 条。
- 消息面：
  - 东方财富新闻搜索事件流。
  - 标题、摘要、媒体、发布时间、链接、事件类型、规则情绪和相关性。
- 汇总层：
  - `stock.analysis.summary.v1`
  - 技术面可为 available。
  - 基本面和消息面当前多为 partial。

Gap：

- 公告 PDF 表格字段抽取未接入。
- 审计意见、完整三表明细、成长趋势、多期现金流分析还不完整。
- 消息面影响强度模型、公告全文解析、LLM 情绪复核尚未接入。
- 技术面建议仍是观察结论，不是完整买卖模型。

瓶颈：

- 基本面和消息面的 provider 多为页面解析，稳定性和维护成本高。
- 正式结论需要事实集、模型和缓存共同支撑，不能每次实时拉取所有源。

## 7. AI 选股与策略回测

预期功能：

- 支持多种选股策略。
- 支持全 A 股样本池。
- 根据策略筛选候选股。
- 用最近几天数据测不同策略胜率。
- 胜率结论要有样本量、基准、超额收益、置信区间和可信度。
- 回测结果可追溯到策略版本、参数、行情快照和回测批次。

已实现：

- AI 选股默认通过 Sina `hs_a` 获取全 A 股样本池，约 5500 只。
- 当前内置 3 个策略：
  - `A杀后横盘放量`
  - `放量突破平台`
  - `跌破后收复关键均线`
- 支持自然语言参数：
  - `扫描上限=50`
  - `验证天数=5`
  - `持有天数=3`
  - `回撤阈值=18`
  - `横盘振幅=10`
  - `量比阈值=1.5`
- `strategyTournament` 已实现：
  - 同批 K 线比较三类策略。
  - 统计信号数、胜率、平均收益、最好/最差收益、当前命中数。
  - 生成 `batchId`。
  - 每个策略写入 `Backtest / BacktestResult`。
  - `reviewReportJson` 保存查询、参数、样本、数据质量、信号和候选。
- 可信度增强已实现：
  - 同窗口全样本基准收益。
  - 策略超额收益。
  - 样本充分度。
  - 95% Wilson 胜率置信区间。
  - 可信评级：high / medium / low / insufficient。
- 最近一次 `扫描上限=50` 验证：
  - 基准样本 250。
  - 基准平均收益 -2.95%。
  - 基准胜率 21.2%。
  - 三类策略均为 low 可信。

Gap：

- 全 A 同步扫描不可接受。
- 策略数量少，仍是确定性规则策略。
- 未接入外部策略发现、策略导入审核、参数优化。
- 未组合投资执行策略，例如分批买入、止损、止盈、移动止盈、仓位控制。
- 未展示收益曲线、回撤曲线、策略热力图、参数敏感性图。
- 未加入费用、滑点、涨跌停、停牌、成交量约束。

瓶颈：

- 全 A 扫描当前同步 API 会尝试拉取约 5500 只股票历史 K 线，实际运行超过 6 分钟仍无结果，必须改成异步 Operation。
- 历史 K 线缺少落地缓存和增量更新。
- 外部行情 provider 吞吐和稳定性不足，无法支撑每次实时全量扫描。

## 8. Operation / 任务中心 / MCP Connect

预期功能：

- 长耗时任务异步化。
- 所有重要任务有 operation_id、状态、进度、artifactRefs、nextActions。
- FAMS 未来可通过 MCP / DomainPack 暴露给外部 Agent 或 harnessOS。

已实现：

- Operation 已用于价格刷新、建议、回测、告警检查。
- 任务中心可查看 Operation 详情和 artifactRefs。
- V2.0 FAMS Connect 与 MCP 工具契约已完成第一轮：
  - HTTP MCP Bridge。
  - stdio MCP Provider。
  - DomainPack manifest。
  - 工具调用 envelope。
  - 用户上下文边界。
  - 交易写入人工确认阻断。
  - harnessOS connector manifest。

Gap：

- AI 选股全 A 扫描还没有 Operation 化。
- 策略锦标赛、全量分析、批量事实集生成都需要后台任务和进度查询。
- 任务产物和前端可视化还需要统一。

瓶颈：

- 同步 API 不适合全市场扫描。
- 当前进程内任务 registry 不适合长期任务和服务重启恢复，需要更持久的任务队列或数据库状态机。

## 9. 当前最大瓶颈排序

1. 全 A 历史 K 线扫描同步执行不可接受。
   - 现象：全 A 选股同步请求超过 6 分钟无结果。
   - 需要：异步 Operation、进度、取消、结果产物、失败重试。

2. 历史行情缓存缺失。
   - 现象：每次全量策略评估都重新拉外部 K 线。
   - 需要：本地 K 线表、增量更新、provider 健康状态、缓存命中率。

3. 外部 provider 稳定性不足。
   - 现象：axios 经 WSL 代理访问部分接口会 400/502/timeout。
   - 需要：多 provider、curl fallback、速率限制、熔断、批量拉取。

4. 策略模型仍偏规则化。
   - 现象：当前 3 个策略在 `扫描上限=50` 短窗验证中均为 low 可信。
   - 需要：更多策略、参数空间、执行策略组合、费用滑点、长窗口验证。

5. 股票分析事实集尚未完全闭环。
   - 现象：基本面为 partial，消息面为 partial。
   - 需要：PDF 表格抽取、公告全文解析、影响强度模型、完整三表和多期趋势。

6. 持仓研究未自动嵌入完整事实集。
   - 现象：持仓面板只展示研究边界，完整个股事实集仍需进入股票分析页。
   - 需要：后台批量生成持仓事实集缓存。

## 10. 想向 ChatGPT 咨询的问题

可以直接复制以下问题：

```text
我在开发一个个人投资资产管理与分析系统 FAMS。它已经有资产/持仓/交易/行情刷新/告警/分析建议/股票三面事实集/AI 选股/短窗回测/MCP 工具契约。

当前目标不是自动交易，而是高可靠、高正确：标的身份、行情、净值、持仓、盈亏、分析证据、选股胜率都必须可追溯。

目前最大问题是：AI 选股全 A 股样本池约 5500 只股票，同步拉取历史 K 线并计算策略胜率超过 6 分钟仍无结果。系统需要改造成异步 Operation，并增加本地历史 K 线缓存、增量更新、provider 健康状态和策略回测产物。

请从架构角度帮我评审：
1. 历史行情缓存表应该如何设计？按 symbol/date/provider 存 OHLCV 是否足够？
2. 全 A 选股扫描应该如何拆成任务、分片、进度、重试和取消？
3. 如何设计 provider 熔断、速率限制和多源交叉验证？
4. 当前策略胜率评估已有基准收益、超额收益、Wilson 置信区间和可信评级，还缺哪些统计指标？
5. 如何把选股策略和投资执行策略组合成可扩展的策略锦标赛？
6. 如何让持仓研究面板批量使用股票事实集，同时不拖慢前端？
7. 在不做自动交易的前提下，下一阶段最应该优先实现哪些功能？
```

## 11. 建议下一步

优先级建议：

1. 将 AI 选股全 A 扫描改成异步 Operation。
2. 新增历史 K 线缓存和增量更新。
3. 全 A 扫描按批次分片，记录每片成功/失败/耗时/provider。
4. 策略锦标赛结果前端可视化：排行榜、基准对比、置信区间、样本分布、交易样本明细。
5. 持仓事实集后台批处理，前端只读缓存结果。
